Proje Karar Özeti 1. Projenin amacı Kurulacak sistem: ticari olarak satılabilir güvenli modüler büyüdükçe bozulmayan çok kullanıcılı kullanıma uygun üretim, montaj, ofis operasyonu ve finans takibini aynı çatı altında yöneten bir web uygulama olacak. Bu proje basit bir iş takip ekranı değil; sipariş merkezli çalışan operasyon ve finans yönetim sistemi olacak. 2. Temel mimari ilkeler Değişmez kurallar kullanıcı girişi güvenliği tavizsiz olacak veritabanı değişiklikleri yalnızca migration ile yapılacak sistem baştan büyümeye uygun tasarlanacak dosya yapısı modüler olacak devasa tek dosya mantığı olmayacak para hesaplarında float kullanılmayacak kritik kayıtlar sessizce değiştirilmeyecek Kod standardı route / service / repository ayrımı olacak tek sorumluluk ilkesi uygulanacak küçük dosya politikası olacak ideal dosya boyu 150–300 satır 400 satır üstü dikkat 500+ satır bölünmeli 700+ satır mimari alarm 3. Teknoloji yaklaşımı Backend FastAPI PostgreSQL SQLAlchemy 2.x style Alembic Pydantic Frontend ayrı frontend mimarisi önerilen yapı: Next.js + TypeScript + Tailwind API-first yaklaşım Neden bu yapı? ürünleşmeye uygun rol bazlı ekran yönetimine uygun modern panel yapısına uygun ileride mobil / farklı istemcilere açık 4. Güvenlik kararları Parola güvenliği parolalar düz metin tutulmayacak hash olarak saklanacak modern ve güvenli parola hash yaklaşımı kullanılacak Giriş güvenliği başarısız giriş denemeleri loglanacak brute-force koruması olacak rate limit / geçici kilitleme olacak IP / user-agent / zaman damgası kaydı tutulacak şüpheli girişler kayda alınacak Oturum / token yapısı access + refresh mantığı olacak token yaşam döngüsü kontrollü olacak kritik roller için ileride 2FA altyapısına uygun tasarım yapılacak Audit güvenlik logları tutulacak kritik işlemler ayrıca loglanacak 5. Yetkilendirme modeli Nihai karar Role + Permission birlikte kullanılacak. Sadece rol kontrolü yapılmayacak. Gerçek yetki kontrolü permission bazlı olacak. Roller Başlangıç rol seti: super_admin owner operation_manager production_master installation_team Önemli kararlar bir kullanıcı birden fazla role sahip olabilir rollerden gelen izinler birleşir en az yetki ilkesi uygulanır tenant / kapsam kontrolü ayrıca uygulanır Rol bazlı genel görünüm super_admin tüm sistem owner şirketin tüm operasyon ve finans görünürlüğü tüm yetkiler operation_manager finansal veriler stok takibi atanmış işler teslim süreci operasyonel kontrol production_master finans hariç üretim odaklı kullanım installation_team saha / montaj ekranları müşteri adres ve iletişim bilgilerine erişim 6. Çok kullanıcılı kullanım ve çakışma yaklaşımı Nihai karar Sistem aynı anda birden fazla kişi tarafından kullanılabilecek. Yetkiler farklı olacağı için çakışma ihtimali düşük olabilir, ama kritik akışlarda önlem alınacak. Kurallar kritik işlemler transaction içinde çalışacak yarım kayıt oluşmayacak belge numarası üretimi güvenli olacak durum geçişleri kontrol edilecek unique constraint ve DB kuralları kullanılacak gerekli yerlerde hafif concurrency koruması olacak 7. Migration ve veritabanı disiplini Nihai karar her veritabanı değişikliği migration ile yapılacak prod veritabanında elle kolon açma/kapama yapılmayacak migration dosyaları açıklamalı olacak migration review yapılacak upgrade öncesi kontrol olacak staging mantığı desteklenecek 8. Dosya / belge yönetimi Nihai karar dosyaların kendisi veritabanında tutulmayacak dosya metadatası veritabanında tutulacak dosyalar storage katmanında saklanacak Dosya sistemi özellikleri tüm ana kayıtlara bağlanabilmeli klasörleme / etiketleme desteği olacak dosya erişimi yetki ve kapsam kontrolü ile olacak Desteklenecek dosya türleri görsel PDF telefon fotoğrafları dekont sözleşme teklif formu 9. Soft delete / archive politikası Master data Soft delete veya aktif/pasif mantığı uygulanabilir: users customers suppliers materials roles Hareket kayıtları Gerçek silme olmayacak: audit_logs login_attempts finansal hareketler stok hareketleri kapanış kayıtları onaylanmış belgeler Gerektiğinde iptal / ters kayıt / düzeltme mantığı kullanılacak. 10. API standardı Nihai karar API versiyonlu olacak api/v1 ile başlanacak response yapısı kontrollü olacak hata formatı standart olacak breaking change olursa yeni versiyon açılacak 11. Config / environment standardı Nihai karar tüm config environment tabanlı olacak secret’lar repoda olmayacak development / staging / production ayrımı olacak eksik kritik env varsa uygulama açılmayacak startup aşamasında config validation olacak 12. Domain event / notification yaklaşımı Nihai karar event odaklı düşünce yapısı olacak ilk sürümde hafif uygulanacak ileride queue / job / notification sistemine genişleyebilecek 13. Veri doğrulama katmanları Nihai karar 3 katmanlı doğrulama olacak: Schema validation tip format required alanlar Business validation iş kuralı kontrolleri Database validation unique FK constraint not null 14. Seed / bootstrap standardı Nihai karar İlk kurulum seed mantığı olacak. Önerilen seed yapısı: permissions roles role_permissions admin user demo data Kural seed işlemleri idempotent olacak prod ve demo seed ayrılacak 15. Observability / logging İlk sürümde olacaklar uygulama logları güvenlik logları hata logları kritik iş logları yavaş endpoint takibi için altyapı düşüncesi 16. Tenant / veri erişim sınırı Nihai karar Sistem çok kiracılı büyümeye hazır olacak. Hazırlık alanları: company_id gerekirse branch_id Kural Bir kullanıcı izin sahibi olsa bile sadece kendi kapsamındaki veriyi görecek. Yani: permission yeterli değil ayrıca tenant / kapsam kontrolü var 17. State machine yaklaşımı Nihai karar State machine olacak ama sade tutulacak. Çok detaylı, kullanıcıyı boğan akışlar yapılmayacak. 18. Belge numaralandırma standardı Nihai karar Belgeler benzersiz numaraya sahip olacak. Örnek yapı: Q-2026-03-0001 O-2026-03-0001 P-2026-03-0001 M-2026-03-0001 Kural aynı işe ait belgelerde benzer numara mantığı olacak yıl / ay bazlı mantık desteklenecek tekrar kullanılmayacak güvenli üretilecek 19. Kullanılabilirlik ilkesi En kritik kararlarından biri buydu: Ana ilke İçeride güçlü mimari, dışarıda sade kullanım Yani: sistem arka planda güçlü olacak kullanıcıdan minimum veri istenecek ilk ekranda sadece gerçekten gerekli alanlar olacak detay alanlar sonradan açılacak süreç fazla detaylandırılıp kullanıcı bıktırılmayacak Resmi kullanılabilirlik kararları veri girişi minimum tıklama ile yapılacak zorunlu alan sayısı minimum tutulacak detay alanlar opsiyonel / ikinci katman olacak rol bazlı sade ekranlar olacak aynı bilgi tekrar tekrar girdirilmeyecek 20. İş akışı ana modeli Kişiye özel işlerde ana akış Müşteri Talebi → Keşif / Ölçü → Malzeme Seçimi + Yaklaşık Maliyet → Teklif → Müşteri Onayı → Manuel Sipariş → Üretim ve/veya Montaj → Teslim Nihai karar teklif onaylanmadan süreç başlamaz teklif onaylandıktan sonra sipariş manuel oluşturulur sipariş oluşturulduktan sonra üretim / montaj / teslim süreci başlar 21. Sistemin ana kaydı Nihai karar Ana kayıt = Sipariş Yani sistemin kalbi order olacak. Sipariş altında bağlanacak yapılar: müşteri bilgisi teklif ilişkisi üretim görevi montaj görevi notlar dosyalar finans hareketleri / özet durum geçmişi 22. Teklif süreci Nihai karar teklif zorunlu teklif onaylanmadan siparişe geçilmez teklif onaylandıktan sonra sipariş manuel açılır tekliften siparişe veriler taşınır Tekliften siparişe taşınacaklar müşteri bilgisi teknik notlar teklif tutarı seçilen malzemeler tahmini maliyet tahmini teslim süresi 23. Keşif / ölçü süreci Nihai karar keşif / ölçü / teknik detay süreci vardır ilk sürümde ayrı büyük modül yapılmayacak ama not ve teknik detaylar mutlaka girilecek 24. Sipariş türleri Nihai karar Bir sipariş şu tiplerde olabilir: sadece üretim sadece montaj / uygulama / teslim hem üretim hem montaj Yani sipariş altında üretim ve montaj süreçleri opsiyonel olarak açılabilecek. 25. Atama yapısı Nihai karar ilk ekranda siparişe tek sorumlu atanacak sistem arka planda çoklu atamayı destekleyecek detay atamalar sonradan açılabilecek Bu sayede kullanıcı yorulmayacak ama sistem esnek kalacak. 26. Üretim modülü İlk sürüm üretim ekranı Başlat Bitir Malzeme Seç Not Ekle Nihai karar üretim süreci sade tutulacak detaylı aşama yönetimi ilk sürümde olmayacak ama kullanılan malzemeler mutlaka üretimi yapan kişi tarafından kaydedilecek 27. Montaj modülü İlk sürüm montaj ekranı Planlanan tarih Tamamlandı Fotoğraf yükle Eksik / hata notu Onay belgesi yükle Nihai karar montaj süreci üretimden biraz daha detaylı olacak eksik / hata bildirimi ayrı modül değil, not olarak girilecek 28. Not sistemi Nihai karar tek tip serbest not sistemi olacak herkes tarafından görülebilecek sipariş ve ilgili süreçlerde not girilecek 29. Dosya sistemi detay kararı Nihai karar dosya yükleme sadece basit upload/listeden ibaret olmayacak klasörleme / etiketleme desteği olacak 30. Müşteri bilgilerinin siparişe kopyalanması Nihai karar Sipariş açılırken müşteri adres ve iletişim bilgileri sipariş içine kopyalanacak. Sebep: müşteri kartı sonradan değişirse eski iş bozulmayacak sipariş kendi tarihsel bilgisini koruyacak 31. Finansal kapsam İlk sürümde kesin olacaklar teklif tutarı sipariş tutarı tahmini maliyet gerçekleşen maliyet gider kaydı tahsilat kaydı kâr / zarar görünümü cari hesap mantığı sipariş bazlı finans özeti sipariş bazlı ekstre Nihai karar Sistem tam muhasebe programı olmayacak; ama güçlü bir operasyonel finans özeti ve iş bazlı finans takibi olacak. 32. Kaporo ve ödeme sistemi Nihai kararlar kaporo o işe ait değer olarak tutulmalı aynı müşterinin farklı işleri birbirine karışmamalı her işin ödemeleri kendi içinde takip edilmeli her işin ayrı ekstresi olmalı Teknik karar Kaporo kullanıcıya siparişin ayrı bir değeri gibi gösterilecek. Ama veri modeli açısından ödeme hareketi mantığında da desteklenecek. Yani: siparişte kaporo özeti görünür altta ödeme hareketleri içinde kaporo türü de olabilir 33. Sipariş bazlı ekstre sistemi Nihai karar Her siparişin kendi finansal ekstresi olacak. Ekstrede yer alacaklar: kaporo tahsilatlar ödeme tarihleri ödeme yöntemleri para birimleri açıklamalar gerekirse dekont bağlantıları Müşteri bazında karışık havuz değil, önce iş bazlı ekstre mantığı olacak. 34. Ödeme yöntemleri ve döviz Ödeme yöntemi Desteklenecek: nakit havale kredi kartı Para birimi Desteklenecek: TRY EUR USD GBP Kullanıcı tarafında sembol gösterilebilir ama sistem tarafında standart kod mantığı kullanılacak. 35. Örnek mantık: owner → genel durum, tüm işler, finans operation_manager → operasyon, geciken işler, teslim süreçleri, stok sinyalleri production_master → üretimdeki işler installation_team → atanmış montaj ve teslimatlar 37.
